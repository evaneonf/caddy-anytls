# Caddy AnyTLS Module 产品文档

## 名称

Caddy AnyTLS Module

## 一句话

把 AnyTLS 做成 Caddy 内的能力，让网站和代理共用 443，由 Caddy 自动管理 TLS 证书，AnyTLS 只处理 TLS 解密后的应用层数据，识别失败自动回落到网站。

## 产品目标

这个产品要解决 4 个问题：

1. 网站和代理共用一个公网入口
2. TLS 证书完全交给 Caddy 自动申请和续期
3. AnyTLS 不再作为独立守护进程运行
4. 非 AnyTLS 流量继续正常访问网站，不影响现有 Web 业务

## 非目标

这个产品不做下面这些事：

- 不实现新的代理协议
- 不自己管理证书文件
- 不替代 Caddy 的 HTTP 站点能力
- 不要求用户额外部署 caddy-l4
- 不解决 CDN 兼容性问题
- 不做多协议网关，首版只聚焦 AnyTLS

## 产品定位

这是一个 Caddy 扩展能力，不是独立代理软件。

对用户来说，它应该表现为：

- 继续使用 Caddy 部署网站
- 在同一个 443 上额外启用 AnyTLS
- 用一个域名给网站，一个域名给代理，或在同一站点入口下启用协议识别
- 不再手工维护代理证书

## 核心能力

### 1. 共用 443

- 网站流量和 AnyTLS 流量共用同一个监听端口
- 不要求单独开额外 TLS 端口

### 2. 自动证书

- 证书由 Caddy/CertMagic 自动申请、续期、装载
- 模块不暴露 `cert_file`、`key_file`

### 3. TLS 后识别

- AnyTLS 识别发生在 TLS 握手完成之后
- 模块只处理解密后的应用层字节流

### 4. 网站回落

- 如果连接不是 AnyTLS，自动回落给正常网站
- 用户访问网站不受影响

### 5. 多用户认证

- 支持多个 AnyTLS 用户
- 每个设备或客户端可独立用户名/密码

### 6. 出站转发

- AnyTLS 认证成功后，建立目标连接并转发流量
- 支持基础策略控制

### 7. 热更新

- Caddy reload 后，配置可更新
- 尽量不影响现有网站连接
- 已建立的 AnyTLS 会话在配置卸载/重载时允许被主动终止，以保证新安全策略立即生效

## 用户价值

### 对运维者

- 少一个单独代理进程
- 少一套证书管理
- 少一组额外公开端口
- 网站和代理共用统一入口

### 对最终用户

- 使用标准域名访问代理
- 配置更稳定
- 站点和代理可共存

## 用户场景

### 场景 1：已有 Caddy 网站，希望同机增加 AnyTLS

- 现状：Caddy 已经负责 443
- 目标：不改网站部署方式，增加代理能力
- 结果：网站继续工作，AnyTLS 使用同一入口和证书体系

### 场景 2：不想手工管代理证书

- 现状：独立代理服务要自己配 cert/key
- 目标：让证书自动申请、续期
- 结果：Caddy 接管证书生命周期

### 场景 3：希望代理失败时仍可正常访问网站

- 现状：独立代理服务和网站入口割裂
- 目标：把非代理流量回落给站点
- 结果：普通浏览器访问仍是网站内容

## 工作原理

产品对外表现为一个 Caddy 能力，内部逻辑是：

1. 客户端连接 443
2. Caddy 根据域名完成 TLS 握手
3. TLS 完成后，模块读取最前面的应用层数据
4. 如果判定为 AnyTLS：
   - 进入 AnyTLS 认证和会话流程
5. 如果不是 AnyTLS：
   - 把连接交回正常网站链路
6. 网站按原有 Caddy 路由继续处理

这个产品的关键点在于：

- Caddy 负责 TLS
- 模块负责 AnyTLS
- 网站回退能力是产品的一部分，不是临时补丁

## 对 AnyTLS 的依赖前提

产品依赖 AnyTLS 的一个特性：

- TLS 后的首包有足够早、足够稳定的认证特征，可用于快速识别

因此本产品不是“猜流量类型”，而是“按协议首包做确定性识别”。

## 配置体验

用户应当能以“启用一个模块”的方式使用，而不是理解复杂网络栈。

目标体验：

- 给站点配置域名
- 开启 AnyTLS 模块
- 配置用户列表
- 配置一些安全策略
- 完成

用户不需要：

- 关心证书文件路径
- 单独部署 AnyTLS 服务端进程
- 手工做端口分流

## 配置项原则

应该有：

- 用户列表
- 超时
- 并发限制
- ACL / 私网访问限制
- 日志等级
- fallback 开关
- 目标连接策略

不应该有：

- `cert_file`
- `key_file`
- 自己的 ACME 配置
- 独立 TLS 监听配置

因为这些都应该属于 Caddy 自身。

## 安全要求

首版至少满足：

- 支持多用户，不鼓励共用密码
- 支持单用户禁用/移除
- 默认拒绝访问常见私网地址
- 有连接超时和空闲超时
- 有基本并发限制
- 有认证失败日志
- 不泄露密码明文
- Reload 时不出现明显安全退化
- 用户被禁用后，新命中的 AnyTLS 连接立即拒绝，不回落到网站

## 运维要求

产品上线后要做到：

- 能通过 Caddy reload 更新配置
- 能输出结构化日志
- 能区分网站流量和 AnyTLS 流量
- 能看出认证成功、失败、用户身份、目标地址、连接时长
- 能看出配置卸载导致的 AnyTLS 会话终止
- 最好后续支持指标

## 兼容性要求

首版应明确：

- 依赖 Caddy 的自动 HTTPS
- 依赖域名，不推荐裸 IP
- 兼容已有网站站点
- 不要求替换现有 HTTP 路由
- 不与独立 AnyTLS 进程并存于同一 TLS 入口

## 验收标准

做到下面这些，产品才算成立：

1. 网站可正常继续使用 443
2. AnyTLS 客户端可通过同一个 443 成功连接
3. Caddy 自动申请并续期证书
4. 模块不需要证书文件配置
5. 非 AnyTLS 请求可以回落到网站
6. AnyTLS 认证失败不会把网站打挂
7. 配置 reload 后网站和代理都继续可用
8. 多用户配置可生效
9. 基础日志可用
10. 不需要额外的 caddy-l4

补充说明：

- “继续可用”指 reload 后新配置可继续接入网站和 AnyTLS
- 不要求旧 AnyTLS 会话跨配置代际保活

## 首版范围

首版只做这些：

- Caddy 内启用 AnyTLS
- 复用 Caddy TLS 与证书
- 单端口共存
- 网站 fallback
- 多用户
- 基础策略和日志

## 后续版本

后续再考虑：

- 指标导出
- 更细粒度 ACL
- 用户级限速/限并发
- 管理接口
- 更丰富的审计信息
- 更强的策略路由

## 一句话总结

这个东西要做成一个“嵌在 Caddy 里的 AnyTLS 接入层”：对外像网站的一部分，对内复用 Caddy 的 443 和证书能力，在 TLS 之后识别并接管 AnyTLS 流量，识别失败则无感回落给网站。
